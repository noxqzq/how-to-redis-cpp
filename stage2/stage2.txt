STAGE 2 — Becoming a database (SET / GET / DEL)
===============================================

Where stage 1 left off
----------------------
Stage 1 taught the server to SPEAK — it parses RESP and answers redis-cli in
its own protocol. But it still has no memory: PING and ECHO are pure
request-and-response, and nothing carries over from one command to the next.
Stage 2 adds the thing that makes it a database: state that outlives a
single command.


The core idea: Redis is a map with a protocol in front
------------------------------------------------------
Strip away persistence, expiry, replication, and the exotic data types, and
the heart of Redis is one in-memory dictionary from keys to values. In C++
that is a single line:

    std::unordered_map<std::string, std::string> store;

The interesting question isn't the type — it's WHERE that line goes. Look at
the shape of main():

    declare store          <-- here
    while (true)               accept loop:  one iteration per client
        while (true)           command loop: one iteration per command

The store must sit OUTSIDE both loops. Put it inside the command loop and it
resets on every command; inside the accept loop and it resets whenever a
client disconnects. Outside both, it lives exactly as long as the server
process — so a value SET over one connection is GETtable over a later one.
That placement — one line in the right spot — is the entire "database" part
of this stage.


Two new reply types
-------------------
Stage 1 used +, -, and $. The new commands need the last two tools in the
RESP box (completing the set of five from stage 1):

    :    integer reply       :2\r\n
         DEL answers with how many keys it actually removed. A count is a
         number, not text, so RESP has a numeric type for it. redis-cli
         renders it as (integer) 2.

    $-1  null bulk string    $-1\r\n
         The reply to GET when the key doesn't exist. Note this is NOT the
         empty string — that would be $0\r\n\r\n. RESP deliberately keeps
         "this key holds an empty value" and "this key doesn't exist"
         distinguishable. redis-cli renders the null as (nil).


SET — insert or overwrite
-------------------------
    store[command_parts[1]] = command_parts[2];
    response = "+OK\r\n";

operator[] on a map does insert-or-overwrite in one step, which happens to be
exactly SET's contract: setting an existing key silently replaces the value.
Real Redis answers +OK, so we do too.


GET — and the operator[] trap
-----------------------------
    auto entry = store.find(command_parts[1]);
    if (entry != store.end()) {
        response = encode_bulk_string(entry->second);
    } else {
        response = "$-1\r\n";
    }

You might reach for store[key] here too — don't. On a missing key,
operator[] CREATES the entry (default-constructed empty string) and returns
it. Your read would silently write: GET of a missing key would insert junk
into the database and reply "" instead of (nil). find() looks without
touching — it returns an iterator to the entry, or end() if absent. Rule of
thumb: operator[] when you mean to write, find() when you mean to read.


DEL — the first variadic command
--------------------------------
    int deleted_count = 0;
    for (size_t i = 1; i < command_parts.size(); i++) {
        deleted_count += store.erase(command_parts[i]);
    }
    response = ":" + std::to_string(deleted_count) + "\r\n";

DEL a b c is legal — it takes any number of keys, and replies with how many
of them actually existed. unordered_map::erase(key) conveniently returns
exactly that per key: 1 if it removed something, 0 if there was nothing to
remove. Sum those and you have the reply. This is why the arity check is
">= 2" where SET's is "== 3".


Two kinds of errors
-------------------
Stage 1 had one error path: unknown command. But once commands have real
argument requirements, "I don't know this command" and "I know it, but you
called it wrong" are different failures, and real Redis reports them
differently:

    GET                -> -ERR wrong number of arguments for 'get' command
    GEX foo            -> -ERR unknown command 'GEX'

So each dispatch branch now matches on the NAME first, then checks arity
inside. (Stage 1 folded the arity check into the match — "ECHO && size >= 2"
— which made ECHO-with-no-args fall through to "unknown command 'ECHO'".
Slightly a lie. Now it isn't.) The message is built by a helper:

    std::string wrong_argument_count(const std::string& command_name) {
        return "-ERR wrong number of arguments for '" + command_name + "' command\r\n";
    }


A helper earns its keep
-----------------------
Stage 1 built its one bulk-string reply inline. Now ECHO and GET both need
one, so the format moves into a function:

    std::string encode_bulk_string(const std::string& value) {
        return "$" + std::to_string(value.size()) + "\r\n" + value + "\r\n";
    }

Small, but it's the start of a pattern: as the command table grows, the RESP
encoding knowledge should live in one place, not be re-derived in every
branch.


The full server (stage 2)
-------------------------
See server.cpp — it is stage 1's file plus:
  - the store declaration before the accept loop
  - the SET / GET / DEL branches
  - the two helpers (encode_bulk_string, wrong_argument_count)
  - per-command arity checks
The socket setup, the parser, and the two-loop structure are UNCHANGED. That
is the point of staging: each stage should touch only the layer it's about.


Naming notes (why these names)
------------------------------
    store                  the entire database — the name says what it's FOR,
                           not what it is (map/dict would name the type)
    entry                  what find() hands back: a pointer-like iterator to
                           a key/value entry, or end() meaning "no entry"
    deleted_count          the number DEL will report; "count" makes the +=
                           accumulation read naturally
    encode_bulk_string     verb-first: it turns a value into wire bytes
    wrong_argument_count   reads as the error it produces


Try it with the real client
---------------------------
    redis-cli -p 8080
    127.0.0.1:8080> SET greeting hello
    OK
    127.0.0.1:8080> GET greeting
    "hello"
    127.0.0.1:8080> GET missing
    (nil)
    127.0.0.1:8080> SET greeting goodbye
    OK
    127.0.0.1:8080> GET greeting
    "goodbye"
    127.0.0.1:8080> DEL greeting missing
    (integer) 1
    127.0.0.1:8080> GET greeting
    (nil)

Now the interesting experiment: Ctrl-C out of redis-cli, reconnect, and GET a
key you set earlier — still there. The data's lifetime is the PROCESS, not
the connection. Kill the server and restart it — gone. Making data survive
the process is persistence (RDB/AOF in real Redis), a much later topic.


What to notice
--------------
1. Everything is a string. SET n 42 stores the three characters "4","2" — no
   number type in sight. Real Redis is the same at this level; commands like
   INCR parse the string, add one, and write it back as a string.
2. The value in the map is the same bytes that arrived on the wire, and goes
   back out as-is. Binary-safe for free, because bulk strings are
   length-prefixed — nothing ever scans the value for terminators.
3. One honest limitation still stands from stage 1: parse_command assumes one
   read() == one complete command. Fine for a human at redis-cli; wrong in
   general.


Next: stage 3
-------------
Concurrency. Replace blocking accept/read with an event loop — epoll on
Linux (kqueue is the macOS equivalent; same idea) — so many clients can be
served at once. That rewrite forces the per-connection read buffer anyway,
which also fixes the "one read = one command" assumption for good.
